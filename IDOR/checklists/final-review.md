# 最终追责反查 Checklist

交付前按失败责任标准逐项反查。发现问题必须直接修复报告、补证据或降级结论；不得把缺口写成待办。

## TXT 复刻检查

- [ ] 是否复刻 TXT 的执行边界，而不是只写概括性目标。
- [ ] 是否保留“本地授权测试环境”边界。
- [ ] 是否保留“不得访问公网目标、不得扫描第三方资产、不得读取真实隐私、不得破坏数据库、不得删除业务数据”的禁止项。
- [ ] 是否保留“重点是动态验证，不允许只给静态猜测”。
- [ ] 是否保留暴露面建模的完整范围。
- [ ] 是否保留五类测试身份。
- [ ] 是否保留双租户最小测试数据要求。
- [ ] 是否保留动态验证 10 类方法。
- [ ] 是否保留正向、反向、越界尝试、阻断验证。
- [ ] 是否保留严禁误报 8 条规则。
- [ ] 是否保留高影响判断条件。
- [ ] 是否保留 20 类偏门路径。
- [ ] 是否保留四类结果输出。
- [ ] 是否保留 confirmed 必填字段。
- [ ] 是否保留执行纪律：不编造、not_run、失败原因、candidate 降级。
- [ ] 是否保留反向审计 15 问。
- [ ] 是否保留遗漏路径清单字段。
- [ ] 是否保留非常规路径 15 项补测。

## 可执行性检查

- [ ] `SKILL.md` 是否能直接指导 Claude 执行。
- [ ] 每个阶段是否写清输入、执行、输出、通过标准、失败处理。
- [ ] 是否不存在只有形容词、没有动作和证据要求的句子。
- [ ] 暴露面矩阵字段是否可直接填写。
- [ ] confirmed 模板字段是否可直接填写。
- [ ] 遗漏路径清单是否可直接填写。
- [ ] 非常规路径结果表是否可直接填写。
- [ ] 回归测试脚本要求是否限定本地环境。

## 证据与结论检查

- [ ] 是否所有 confirmed 都有动态请求证据。
- [ ] 是否所有 confirmed 都有资源归属证明。
- [ ] 是否所有 confirmed 都有账号说明。
- [ ] 是否所有 confirmed 都有原始响应关键片段。
- [ ] 是否所有 confirmed 都有代码根因位置或定位依据。
- [ ] 是否所有 confirmed 都有修复建议。
- [ ] 是否所有 confirmed 都有修复后验证方法。
- [ ] 是否所有证据路径真实存在；不存在时是否标记 not_run、blocked 或 failed。
- [ ] 是否没有把空响应误判为越界。
- [ ] 是否没有把管理员正常权限误判为缺陷。
- [ ] 是否没有把测试数据配置错误误判为代码缺陷。

## 架构检查

- [ ] 是否只保留 1 个主 Skill。
- [ ] 是否不存在空壳 Skill。
- [ ] 是否没有无理由新增目录。
- [ ] 是否每个文件都有明确作用。
- [ ] 是否模板可直接填写。
- [ ] 是否 checklist 可验收。
- [ ] 是否 examples 贴近 TXT。
- [ ] 是否 tests 能发现缺漏、摘要化、无关扩展、无门禁、无映射等问题。
- [ ] 是否区分原文复刻和工程化补强。
- [ ] 是否提供 TXT 到 Skill 映射。
