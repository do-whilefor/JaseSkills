二、动态验证总规则

你必须优先动态验证，而不是只读代码。

每个验证必须满足：

1. 在本地启动项目。
2. 使用测试数据库或可回滚事务。
3. 使用测试账号、测试租户、测试对象。
4. 使用唯一 marker，例如：
   INJ_MARKER_<timestamp>_<random>
5. 每个验证必须有：
   - 正向请求
   - 负向对照
   - 被拦截/被转义/被参数化时的正常结果
   - 日志证据
   - HTTP 请求与响应证据
   - 数据库变更前后对比，若涉及数据库
   - 文件变更前后对比，若涉及文件
   - 服务端错误栈，若有
   - 最小复现步骤
6. 禁止使用破坏性 payload。
7. 禁止读取真实敏感数据。
8. 禁止写入项目目录以外的非测试位置。
9. 禁止执行破坏性命令。
10. 命令类验证只允许使用本地无害 marker 输出或写入测试目录 marker 文件。
11. SQL/NoSQL 类验证只允许在测试库、测试表、测试数据中完成。
12. 模板/表达式类验证只允许证明 marker 被解释，不允许执行危险系统调用。
13. 时间类验证如果必要，单次延迟不得超过 1 秒，并且必须证明不是网络抖动。
14. 未经动态复现的结论只能叫 candidate，不能叫 confirmed。

三、注入类型覆盖要求

请至少覆盖以下类型，不要只盯 SQL 注入：

1. SQL 注入
   - raw SQL 拼接
   - ORM raw query
   - order by、group by、limit、offset、field name 动态拼接
   - search filter 动态拼接
   - 多语句配置错误
   - migration、admin console、report query
   - GraphQL resolver 中的动态 SQL
   - 二阶 SQL 注入：先存储，再由报表、搜索、导出、后台任务触发

2. NoSQL 注入
   - Mongo/Redis/Elastic/Solr 查询对象污染
   - JSON operator 注入
   - 类型混淆：字符串、数组、对象、布尔、null
   - 查询 DSL 拼接
   - filter/sort/projection 动态字段
   - GraphQL variables 到 NoSQL query 的转换

3. 命令/参数注入
   - shell=True、exec、spawn、ProcessBuilder、Runtime.exec、child_process
   - 文件转换、图片处理、PDF 生成、压缩解压、备份恢复、git 命令、系统工具调用
   - 参数数组与 shell 字符串边界混淆
   - Windows/Linux 参数解析差异
   - 配置项进入命令行
   - 后台任务触发命令
   - 只允许 marker 级无害验证

4. 模板注入 / 表达式注入
   - Jinja、Twig、Freemarker、Velocity、Handlebars、Mustache、EJS、Pug
   - SpEL、OGNL、MVEL、JEXL、Spring EL、Java EL
   - 邮件模板、通知模板、站内信模板
   - 报表模板、CMS 页面、低代码规则、工作流表达式
   - 管理员预览功能
   - 二阶模板注入：普通用户提交内容，管理员预览时触发

5. XPath / LDAP / XML 注入
   - 登录、搜索、组织架构、权限组、目录查询
   - XML filter 拼接
   - XPath 条件拼接
   - LDAP filter 拼接
   - 只用测试目录和测试数据验证

6. GraphQL 注入
   - resolver 内部 SQL/NoSQL 拼接
   - variables 类型绕过
   - alias/fragments 扩展隐藏路径
   - nested input object 注入
   - sort/filter/order/projection 字段
   - 自定义 scalar 解析

7. Header / CRLF / 响应头注入
   - redirect、download filename、Content-Disposition
   - callback URL、next 参数
   - custom header
   - 日志/审计系统中的换行污染
   - 只验证 header 边界，不做钓鱼或外部跳转攻击

8. JSONPath / JMESPath / Regex / Search DSL 注入
   - 动态构造表达式
   - 用户输入进入过滤表达式
   - 正则灾难性回溯风险只做静态风险提示，不做 DoS 动态打压
   - 搜索语法逃逸和权限过滤绕过

9. Import/Export 注入
   - CSV/Excel/HTML/Markdown/PDF 导出内容
   - 文件名、sheet 名、单元格内容、HTML 模板
   - 只用测试文件和 marker 验证
   - 不触发外部连接，不读取真实敏感文件

10. 二阶注入
   - 注册资料、昵称、备注、地址、订单备注、审批意见、评论、工单、标签、配置项
   - 先存储，再由后台任务、管理员页面、邮件通知、导出、搜索索引、报表生成触发
   - 必须追踪“写入点”和“触发点”

四、偏门与冷门挖掘路线

不要只检查 controller 里的显眼参数。必须额外检查：

1. 隐藏参数：
   - 前端 JS 中出现但 UI 不暴露的参数
   - disabled input
   - localStorage/sessionStorage 中的字段
   - GraphQL schema 暴露但页面不用的字段
   - OpenAPI/Swagger 中未绑定页面的接口
   - 后端 DTO/model 中存在但前端没传的字段

2. 非主流程入口：
   - 管理后台
   - 安装向导
   - 初始化脚本
   - 迁移脚本
   - debug/dev/test endpoint
   - health check
   - webhook receiver
   - import/export
   - report builder
   - notification template
   - workflow/rule engine
   - scheduled jobs
   - CLI tools
   - queue consumers

3. 封装与依赖：
   - 自定义 query builder wrapper
   - 自定义 sanitizer
   - 自定义 template helper
   - 自定义 shell wrapper
   - 第三方库的危险默认配置
   - 框架 magic method / hook / middleware
   - ORM escape 被手动绕开的位置
   - 参数化与字符串拼接混用的位置

4. 多解释器链：
   - 用户输入先进入 Markdown，再进入 HTML，再进入 PDF renderer
   - 用户输入先进入数据库，再进入模板
   - 用户输入先进入 JSON，再进入搜索 DSL
   - 用户输入先进入配置，再进入命令行
   - 用户输入先进入日志，再进入日志查询系统
   - 用户输入先进入队列，再由 worker 执行

5. 类型边界问题：
   - 字符串变对象
   - 数组变字符串
   - null/undefined/空字符串
   - 布尔值变表达式
   - 数字字段传字符串
   - sort/order/filter 传对象
   - content-type 切换导致解析差异
   - URL 编码、双重编码、Unicode 归一化、大小写归一化差异

6. 权限组合：
   - 匿名用户
   - 普通用户
   - 低权限管理员
   - 高权限管理员
   - 租户 A 用户
   - 租户 B 用户
   - 已禁用用户
   - 只读用户
   - API token 用户
   - Webhook 身份

五、严重性判定标准

你必须按证据判定严重性：

confirmed high/critical 必须满足：

1. 有动态请求证据。
2. 有服务端响应或日志证据。
3. 有 marker 被解释、拼接、执行、越权查询或跨边界影响的证据。
4. 有负向对照证明不是正常功能。
5. 有最小复现步骤。
6. 有影响范围说明。
7. 有修复建议。
8. 有误报排除。

不能因为代码里看起来危险就直接标高危。没有动态验证只能标 candidate。
