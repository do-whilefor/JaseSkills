# 最终追责反查清单

## 1. TXT 复刻一致性

- [ ] 保留了本地授权范围。
- [ ] 保留了不扫公网、不访问第三方、不使用真实用户数据。
- [ ] 保留了不做中间人、嗅探、证书劫持、流量劫持、钓鱼、社工。
- [ ] 保留了不做破坏性操作、DoS、压力测试。
- [ ] 保留了测试账号、测试租户、测试角色、测试资源要求。
- [ ] 保留了写操作可回滚要求。
- [ ] 保留了 candidate 与 confirmed 的证据区分。
- [ ] 保留了缺少启动命令、账号、环境变量、DB 初始化脚本时的处理规则。
- [ ] 保留了“先给出将读取的文件清单和认证暴露面建模结果，然后直接进入动态验证”。
- [ ] 保留了“不要停留在静态审计”。

## 7. confirmed 追责反查

对每个 confirmed 逐条检查：

- [ ] 是否有真实动态请求证据。
- [ ] 是否有正向成功样本和反向失败预期样本。
- [ ] 是否有异常成功结果，而不是只凭代码推测。
- [ ] 是否有测试账号、测试资源、测试租户。
- [ ] 是否有状态码、响应体、日志或数据库证据。
- [ ] 是否有 Playwright trace、HAR、截图、curl 或测试用例之一。
- [ ] 是否验证未登录、普通用户、管理员、跨用户、跨租户、禁用用户、未验证用户、退出后旧凭据、过期 token 中相关项。
- [ ] 是否覆盖项目实际存在的 REST、GraphQL、WebSocket、文件下载、导出、异步任务、Webhook、邀请、重置密码、邮箱绑定、OAuth 绑定。
- [ ] 是否检查前端隐藏接口和后端实际接受参数。
- [ ] 是否测试 body/query/path/header/cookie/session 身份字段优先级。
- [ ] 是否测试批量接口、嵌套对象、filter/where/include/expand/fields。
- [ ] 是否确认 200 响应包含敏感内容或产生越权效果。
- [ ] 是否在 403 后检查同资源其他方法、路径、content-type。
- [ ] 是否排除了无法复现线索。
- [ ] 是否检查框架默认路由、fallback、catch-all、静态资源、调试接口。
- [ ] 是否检查中间件顺序、装饰器继承、controller 级和 method 级权限覆盖。
