# Evidence Reporting Quality Gate

这个 Skill 解决的问题：把候选和验证结果转化为可交付报告，同时阻止误报、敏感信息泄露、无证据结论和低质量硬交付。

## 必须调用的场景

- 动态验证结束后准备输出报告。
- 需要分级高、中、低风险发现。
- 需要整理待确认、不可报告或不可交付原因。
- 需要对发现做遗漏反思、误报反思、剑走偏锋补充发现。
- 需要输出下一步最小安全验证清单。
- 用户要求“只给最终结论”但证据链不满足条件时。

## 禁止调用的场景

- 没有动态证据却要求输出确定发现。
- 用户要求完整披露 token、cookie、私钥、密码、连接串、隐私数据。
- 用户要求省略不可报告原因或忽略质量门槛。

## 输入材料

- 资产账本。
- 运行态入口枚举结果。
- 静态候选表。
- 动态验证记录。
- 角色差分矩阵。
- 偏门补漏结果。
- 请求摘要、响应摘要、代码/配置来源、复现次数。
- 文档指纹覆盖表。

## 硬质量门禁

每个“确定发现”必须同时满足以下条件：

| 门禁 | 要求 | 不满足时处理 |
|---|---|---|
| QG-01 授权范围 | 目标属于本机项目、授权目录、授权服务、授权端口、授权账号 | 标记“超出授权范围，未验证” |
| QG-02 动态证据 | 有运行态 URL/端口/路径与请求响应摘要 | 降级为待确认 |
| QG-03 可复现 | 至少复现 2 次，关键发现优先 3 次 | 降级为待确认 |
| QG-04 非预期可见 | 能说明未认证或非预期角色可读，或能说明暴露不应公开 | 管理员预期可见则不可报告 |
| QG-05 实际信息 | 响应包含真实信息内容，不只是空页面、状态码、报错壳 | 降级为待确认或不可报告 |
| QG-06 影响判断 | 有影响对象与安全影响 | 降级为待确认 |
| QG-07 证据脱敏 | 敏感内容只输出脱敏样本、长度、类型、位置、hash | 停止交付并脱敏重写 |
| QG-08 代码/配置关联 | 能关联源码、配置、部署或前端产物来源；无法关联时说明原因 | 降级或标记缺口 |
| QG-09 反证完成 | 已检查测试数据、开发环境、预期权限、缓存偶发、超范围 | 未完成则待确认 |
| QG-10 反思完成 | 已执行遗漏、误报、剑走偏锋反思 | 不允许输出最终报告，只能输出阶段报告 |

## 不可交付原因

出现以下情况时，不要硬写最终报告，只能输出阶段性结果：

- 没有 Base URL 或服务未运行。
- 只有静态命中，没有运行态访问。
- 无法确认目标属于当前项目。
- 没有账号，用户却要求角色差异结论。
- 发现涉及外部真实服务 token 验证。
- 用户要求输出完整敏感数据。
- 关键发现复现次数不足。
- 响应为空、测试数据、管理员预期可见、无实际影响。
- 证据链缺代码/配置/部署来源且无法解释缺口。
- 工具、路径、文件、模板不存在，无法证明。

## 执行步骤

1. 汇总所有候选资产，确保每条都有状态：可报告 / 待确认 / 不可报告 / 不可交付。
2. 为每个证据分配编号：`EV-*` 运行态证据，`SC-*` 静态代码证据，`RD-*` 角色差分证据，`EC-*` 偏门补漏证据，`QG-*` 质量门禁证据。
3. 对所有敏感内容执行脱敏：只保留类型、长度、上下文、文件/接口位置、hash 指纹、脱敏样本。
4. 对每个候选写明“为什么可疑”和“为什么现在能/不能报告”。
5. 对可报告发现写明标题、类型、影响对象、认证状态、角色要求、运行态路径、复现步骤、请求摘要、响应摘要、脱敏证据、代码来源、复现次数、安全影响、风险等级、修复建议。
6. 对待确认项写明缺失证据和下一步最小安全验证。
7. 对不可报告项写明不可报告原因。
8. 对不可交付情况写明阻断项和继续所需最小材料。
9. 执行三轮反思：遗漏反思、误报反思、剑走偏锋补充发现。
10. 输出最终报告或阶段性报告。

## 脱敏规则

- API Key：`sk_live_****abcd`
- JWT：`eyJ****.****.****xyz`
- DB URL：`mysql://user:****@host:3306/db`
- Cookie：`session=****abcd; 长度：xx; SHA256: xxx`
- Private Key：只输出 `类型：Private Key; 长度：xx; SHA256: xxx; 位置：xxx`
- 隐私数据：只输出字段名、数量、样本脱敏、影响对象，不输出完整值。

## 失败处理

- 动态证据不足：降级为待确认。
- 只需管理员权限且属于预期可见：标记不可报告。
- 只有测试数据：标记不可报告，除非运行态真实暴露且有影响。
- 无法稳定复现：标记待确认。
- 超出授权范围：停止验证，标记未验证。
- 缺少硬质量门禁：输出阶段性报告，不输出最终确定结论。

## 与其他 Skills 的协作

- 汇总 01 到 06 的所有结果。
- 是最终质量门槛和报告出口。
- 发现质量门禁不满足时，回调对应 Skill：范围问题回 01，入口问题回 02，静态来源问题回 03，动态证据问题回 04，角色问题回 05，偏门覆盖问题回 06。

## 报告自动评分器增强

基于文档延伸：最终报告或阶段报告生成前，必须对每个发现按 QG-01 到 QG-10 输出通过、失败或未知原因。可使用：

```bash
python3 scripts/qg-finding-score.py draft-report.md -o qg-finding-score.md
```

评分结果处理：任一 QG 为 FAIL，该发现不可交付或不可报告；任一关键 QG 为 UNKNOWN，该发现降级为待确认；全部 PASS 只能说明证据结构满足门槛，仍需人工复核。

新增输出要求：最终报告增加 `## I. 每个发现的 QG 评分`。如果没有执行评分器，必须手工按同一字段输出评分，不得省略。


## 质量门禁补丁：逐发现评分与不可报告决策树

基于文档延伸：最终报告前必须同时满足两层检查。

第一层：使用 `templates/non-reportable-decision-tree.md` 判断每个候选是否可报告、待确认、不可报告或不可交付。

第二层：使用 `scripts/qg-finding-score.py` 或同等手工表，对每个发现按 QG-01 到 QG-10 评分：

- 任一 FAIL：不可交付或不可报告，必须修复或降级。
- 任一关键 UNKNOWN：降级待确认，并写下一步最小补证。
- 全 PASS：只代表结构满足门槛，仍需人工复核事实真实性。

报告中必须包含 `## I. 每个发现的 QG 评分` 和 `## H. 证据链压缩摘要`。缺少这两项时，只能输出阶段性报告。

## Manifest 与自测门禁

最终报告前应执行两个附加检查：

1. 证据清单生成：使用 `scripts/report-to-manifest.py` 把资产账本、请求摘要、响应摘要、截图、代码证据、QG 评分统一写入 evidence manifest。Manifest 只作为证据索引，不自动证明漏洞成立。
2. Skill 自测：维护或发布前使用 `scripts/skill-selftest.py` 检查所有 SKILL.md 是否包含必须调用、禁止调用、输入、步骤、输出、失败处理、质量门槛、协作、反注入、可追溯、延伸标记。

若 manifest 缺少动态证据、QG 评分或代码来源，相关发现必须降级为待确认/不可报告。若自测失败，应修复 Skill 文档后再发布。

## 工程化补丁：Manifest、Index、Dashboard 验收

新增验收入口：

```bash
python3 scripts/evidence-schema-validate.py evidence-manifest.json --kind evidence-manifest
python3 dashboard/dashboard_generator.py --root . --selftest-dir selftest/out --out selftest/out/dashboard.html
bash scripts/run-package-selftest.sh . selftest/out
```

新增硬规则：

1. `manifests/template_index.json` 中声明的模板必须真实存在，否则不可发布。
2. `manifests/knowledge_index.json` 中声明的知识条目必须真实存在；冲突和待确认条目必须进入 `human_review/human_review_queue.jsonl`。
3. `dashboard/dashboard_generator.py` 只能读取真实 selftest 输出，不能生成伪成功结果。
4. `scripts/evidence-schema-validate.py` 是 stdlib 必填字段检查，不等同于完整 JSON Schema validator。若需要完整 JSON Schema 校验，必须在本机明确安装并记录 validator 版本。
5. dashboard 显示 `Route → Candidate → Evidence → Quality Gate → Report` 链路时，必须保留 “candidate is not finding” 限制。
