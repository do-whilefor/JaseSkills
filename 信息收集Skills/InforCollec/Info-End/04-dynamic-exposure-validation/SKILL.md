# Dynamic Exposure Validation

这个 Skill 解决的问题：判断候选信息面是否“现在真的从运行态可访问、真的包含非预期信息、真的有影响、真的可复现”。

## 必须调用的场景

- 存在来自运行态枚举或静态分析的候选 URL、接口、文件、文档、debug、metrics、source map、错误面。
- 用户要求验证信息暴露是否存在。
- 需要区分工具命中、静态线索、异常响应与真实信息暴露。

## 禁止调用的场景

- 目标不在授权范围内。
- 用户要求危险 payload、破坏性测试、数据修改、压测、爆破。
- 用户要求用真实泄露凭证调用外部服务。
- 用户要求完整输出敏感数据。

## 输入材料

- Base URL。
- 候选路径或接口表。
- 认证上下文：未认证、低权限账号、普通用户、管理员等。
- 静态证据位置。
- 禁止范围。

## 执行步骤

1. 对每个候选先确认是否在授权范围内。
2. 未认证视角访问候选路径。
3. 如有授权账号，再做低权限和其他角色访问。
4. 对安全请求方法做差异观察：HEAD、GET、OPTIONS。
5. 对正常 Accept 与异常 Accept 做差异观察，例如 `application/json` 与 `text/html`，但不注入危险 payload。
6. 比较正常路径与尾斜杠路径。
7. 比较前端入口访问与 API 直接访问。
8. 比较浏览器访问与 curl/HTTP 客户端访问。
9. 清 Cookie、登出或刷新后再次访问，排除缓存和旧 session 干扰。
10. 至少重复验证 2 次；高风险或不稳定发现优先 3 次。
11. 记录请求摘要、响应摘要、状态码、响应头、内容类型、响应大小、敏感字段类型、认证要求、角色要求、复现次数。
12. 将无法满足证据门槛的候选降级为“待确认 / 不可报告”。


## 单项输入-处理-输出闭环

每个候选必须按同一格式执行：

| 字段 | 要求 |
|---|---|
| 输入 | 候选编号、来源 Skill、Base URL、路径、方法、认证上下文、静态证据 |
| 处理 | 范围确认 → 只读请求 → 响应摘要 → 敏感字段脱敏 → 复现 2-3 次 → 反证检查 |
| 输出 | 动态证据编号、请求摘要、响应摘要、认证要求、角色要求、是否敏感、是否可报告、不可报告原因 |

如果无法完成任一字段，候选不得升级为确定发现。

## 检查点

- 是否真的访问到了运行态路径？
- 是否确认内容非空且有实际信息？
- 是否确认不是测试数据或公开预期信息？
- 是否确认是否需要认证？
- 是否确认是否被非预期角色访问？
- 是否至少复现 2 次？
- 是否有代码或配置来源？
- 是否脱敏？
- 是否写明可报告或不可报告原因？

## 质量门槛

- 工具告警不是结论。
- 状态码变化不是结论。
- 报错页面不是结论。
- 关键词命中不是结论。
- 无动态证据不得报告。
- 无影响判断不得报告。
- 无法稳定复现不得报告。
- 管理员预期可见内容默认不作为信息泄露。
- 所有敏感数据必须脱敏。

## 失败处理

- 401/403：记录认证需求，不直接判定安全或不安全。
- 404/405：记录错误体、允许方法、响应头，但不直接报告。
- 500：只作为错误面候选，需确认是否泄露栈、路径、schema、服务信息。
- 空响应：分析是无数据、权限过滤还是参数缺失，不能直接下结论。
- 访问跳转到外部域：停止跟随，记录超出范围。

## GraphQL 非破坏验证增强

基于文档延伸：当候选入口是 GraphQL 时，必须遵循 `templates/graphql-nondestructive-validation.md`。

允许的最小验证：OPTIONS、`{ __typename }`、格式错误响应、不存在字段错误富信息、不同角色同请求对比。默认禁止 introspection、mutation、字段爆破和大规模枚举。

GraphQL 结果仍需满足 QG-01 到 QG-10，错误体信息不能直接当确定发现。


## 动态验证补丁：变体、协议和回流

基于文档延伸：动态验证必须显式覆盖原文中的请求方法、Accept、尾斜杠、前端/API、浏览器/curl、刷新/登出/清 Cookie、重复验证 2-3 次。

可使用 `scripts/exposure-probe-safe.sh`，其输出包含：method、url、variant、accept、run、status、content_type、size、location、server、allow、cache_control。

协议差异参考 `templates/protocol-surface-validation.md`：WebSocket/SSE/gRPC/RPC 只做非破坏可达性、认证要求、错误摘要和元数据记录；不得长连接滥用、爆破或调用修改方法。

GraphQL 结果必须按四类回流：错误富信息、schema 影子线索、角色差异、不可报告原因。只看到 `introspection disabled` 且无额外信息时不可报告。

## gRPC/RPC schema-aware 只读验证

当候选为 gRPC/RPC 时，必须使用 `templates/grpc-rpc-schema-aware-readonly-validation.md`。默认只允许：

1. 本地 schema/IDL/proto/文档读取。
2. reflection list/describe 类元数据验证。
3. HTTP/RPC endpoint 的 OPTIONS、错误格式、Content-Type、错误扩展观察。
4. 用户明确 allowlist 的只读、无副作用方法。

禁止默认调用 create/update/delete/send/reset/invite/import/export 等可能改变状态的方法。无法证明只读时，状态写为 `pending_confirmation` 或 `non_reportable`，不可硬报。

04 回填资产账本时必须符合 `schemas/asset-ledger.schema.json`，并保留动态证据编号。
