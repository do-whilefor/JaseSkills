# 质量门禁

任何确定发现必须通过 QG-01 到 QG-10。未通过时只能进入待确认、不可报告或不可交付。

| 编号 | 门禁 | 通过条件 | 失败处理 |
|---|---|---|---|
| QG-01 | 授权范围 | 本机项目、授权目录、授权服务、授权端口、授权账号 | 超出授权范围，未验证 |
| QG-02 | 动态证据 | 有运行态 URL/端口/路径和请求响应摘要 | 待确认 |
| QG-03 | 可复现 | 至少复现 2 次，关键发现优先 3 次 | 待确认 |
| QG-04 | 非预期可见 | 未认证或非预期角色可读，或暴露不应公开 | 不可报告或待确认 |
| QG-05 | 实际信息 | 响应含真实信息，不只是状态码/工具告警 | 待确认/不可报告 |
| QG-06 | 影响判断 | 有影响对象、安全影响 | 待确认 |
| QG-07 | 脱敏 | 敏感内容已脱敏，仅含样本、长度、类型、位置、hash | 停止交付并脱敏重写 |
| QG-08 | 来源关联 | 有代码/配置/部署/前端产物来源，或说明无法关联原因 | 缺口记录 |
| QG-09 | 反证 | 已排除测试数据、开发环境、管理员预期可见、缓存偶发、超范围 | 待确认 |
| QG-10 | 三轮反思 | 遗漏、误报、剑走偏锋已执行 | 只能输出阶段报告 |

## 不可交付输出格式

```md
# 不可交付原因
- 阻断门禁：
- 当前已有证据：
- 缺失证据：
- 为什么不能给确定结论：
- 下一步最小安全验证：
```

##

质量门禁是对原文“质量门槛”和“不可报告原因”的工程化，不改变原文结论标准。

## QG 自动评分器

基于文档延伸：可使用 `scripts/qg-finding-score.py` 对报告中每个发现按 QG-01 到 QG-10 输出 PASS / FAIL / UNKNOWN。

- FAIL：该发现不可交付或不可报告，必须修复。
- UNKNOWN：证据不足，降级待确认。
- PASS：只代表证据结构满足门槛，仍需人工复核。

评分器不能替代动态验证、角色差分、反证和人工判断。
