# 质量门禁清单

## 1. 交付前基础门禁

- [ ] 已读取原始 TXT。
- [ ] 已读取当前 Skill 全部文件。
- [ ] 已确认只有 1 个主 Skill。
- [ ] 已确认目录名为 `business-logic-verify`，小写英文短横线命名。
- [ ] 已确认不存在 best、final、new、advanced、ultimate、audit-skill 等空泛命名。
- [ ] 已确认不存在空文件、重复文件、无理由目录。
- [ ] 已确认每个文件能追溯到 TXT 原文或工程化补强理由。
- [ ] 已确认工程化补强没有伪装成 TXT 原文。

## 2. 范围与安全门禁

- [ ] 目标是当前本地授权开源项目。
- [ ] 验证环境是本地测试环境。
- [ ] 使用测试数据库或可回滚快照。
- [ ] 使用测试账号、测试租户、测试订单、测试资金、测试积分、测试优惠券、测试邀请码、测试 webhook。
- [ ] 不访问公网敏感地址、内网敏感地址、云元数据地址。
- [ ] 不调用真实支付、真实 webhook、第三方生产系统。
- [ ] 不删除真实数据、不破坏数据库、不影响真实业务。
- [ ] 不做 DoS、DDoS、资源耗尽、批量压测、恶意刷接口。
- [ ] 所有验证步骤可回滚。

## 3. 项目理解门禁

- [ ] 已识别语言、框架、入口文件。
- [ ] 已识别路由、控制器、service、model、middleware。
- [ ] 已识别 worker、队列、定时任务、background job。
- [ ] 已识别数据库 schema、关键表、关键状态字段。
- [ ] 已识别 session、JWT、OAuth、API key、magic link、邮箱验证码、手机验证码、webhook signature、admin token、service token 中项目实际存在项。
- [ ] 已建立项目实际存在的角色矩阵。
- [ ] 多租户项目已建立 tenant A 和 tenant B。
- [ ] 已识别项目实际存在的核心业务对象。
- [ ] 已识别项目实际存在的状态机。
- [ ] 已输出正常、异常、取消、重试、回滚、管理员干预、webhook 异步回调、队列补偿流程。

## 4. 十类业务逻辑覆盖门禁

- [ ] 权限错位：低权限、高权限、跨用户、跨租户、禁用/移除/降权后权限残留已验证或标注不适用。
- [ ] 状态机绕过：正常状态流和非法状态跳转已验证或标注不适用。
- [ ] 金额、余额、积分、优惠：负数、小数、边界、叠加、重复、跨用户/跨租户、过期、回滚、权益残留已验证或标注不适用。
- [ ] 幂等和重复提交：相同请求、idempotency key、缺失 key、webhook 重放、唯一约束、事务、状态锁已验证或标注不适用。
- [ ] 并发竞争：仅使用 2 到 5 个并发请求；已记录每个响应和最终 DB 状态。
- [ ] Webhook 和异步回调：使用本地 mock webhook；验证签名、timestamp、event_id、金额、租户、worker 日志。
- [ ] 账号生命周期：注册、邮箱、OAuth、重置密码、旧 session/token/API key、删除、禁用、移除、邀请已验证或标注不适用。
- [ ] 后端接受隐藏参数：从正常请求增加敏感字段并对比 DB。
- [ ] 多租户隔离：导入导出、报表、邀请、优惠券、管理后台、缓存 key、queue topic、worker、webhook tenant context 已验证或标注不适用。
- [ ] 业务规则反推：已列出核心业务不变量，并为每条不变量设计 replay plan。

## 5. Replay plan 门禁

- [ ] 每个候选点都有编号。
- [ ] 每个候选点写明业务对象。
- [ ] 每个候选点写明正常业务流程。
- [ ] 每个候选点写明被测试的不变量。
- [ ] 每个候选点写明角色、租户、数据状态、token/session、业务对象 id。
- [ ] 每个候选点写明触发入口：HTTP API、GraphQL、WebSocket、CLI、webhook、worker、admin panel、import/export、background job。
- [ ] 每个候选点写明请求构造方式。
- [ ] 每个候选点写明正向验证。
- [ ] 每个候选点写明负向验证。
- [ ] 每个候选点写明证据需求。
- [ ] 每个候选点写明计划结论。

## 6. confirmed 证据门禁

- [ ] 每个 `confirmed` 都有明确被破坏的业务不变量。
- [ ] 每个 `confirmed` 都有触发请求。
- [ ] 每个 `confirmed` 都有后端响应。
- [ ] 每个 `confirmed` 都有触发前数据库状态。
- [ ] 每个 `confirmed` 都有触发后数据库状态。
- [ ] 每个 `confirmed` 都有关键日志。
- [ ] 每个 `confirmed` 都有相关代码位置。
- [ ] 每个 `confirmed` 都证明低权限或异常状态完成了本不应成功的业务动作。
- [ ] 每个 `confirmed` 至少有一个负向验证样本。
- [ ] 每个高危 `confirmed` 都有修复后回归测试。

## 7. 禁止误判门禁

- [ ] 没有把只有代码片段的问题标为 `confirmed`。
- [ ] 没有把只有 success 响应、无业务状态变化的问题标为 `confirmed`。
- [ ] 没有把只有前端异常的问题标为 `confirmed`。
- [ ] 没有把理论推断标为 `confirmed`。
- [ ] 没有把环境配置问题误判为业务漏洞。
- [ ] 没有把管理员正常能力误判为越权。
- [ ] 没有把测试接口、mock 接口、开发接口误判为真实运行路径。
- [ ] 没有把失败响应误判为漏洞。
- [ ] 没有忽略 401/403。
- [ ] 没有把 `candidate` 伪装成 `confirmed`。
- [ ] 没有把 `blocked` 路径误判成漏洞。

## 8. 输出门禁

- [ ] 输出包含项目业务暴露面总览。
- [ ] 输出包含已确认业务逻辑漏洞。
- [ ] 输出包含候选漏洞。
- [ ] 输出包含被阻断路径。
- [ ] 输出包含 false_positive。
- [ ] 输出包含 needs_environment。
- [ ] 输出包含工程级修复建议。
- [ ] 输出包含第一轮强制反思。
- [ ] 输出包含第二轮偏门补挖。
- [ ] 输出包含最终 confirmed 审判。
- [ ] 输出包含证据索引。
- [ ] 输出不可追溯证据编号时不得交付。
