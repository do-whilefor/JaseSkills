# 最终追责反查清单

## 1. 每个 confirmed 的 20 项审判

对每一个 `confirmed` 单独填写。任一关键项失败，必须降级为 `candidate`、`blocked`、`false_positive` 或 `needs_environment`。

| confirmed 编号 | 标题 | 当前严重性 | 审判后分级 | 降级原因或保留理由 |
|---|---|---|---|---|
| `<BLV-*>` | `<title>` | `<severity>` | `<confirmed/candidate/blocked/false_positive/needs_environment>` | `<reason>` |

- [ ] 是否真的破坏了明确的业务不变量。
- [ ] 是否有动态请求和响应。
- [ ] 是否有数据库前后状态对比。
- [ ] 是否有日志或 worker 处理证据。
- [ ] 是否证明业务结果真的发生，而不是只返回 success 文案。
- [ ] 是否证明低权限角色可以完成本不应完成的动作。
- [ ] 是否证明跨租户影响，或者明确没有跨租户影响。
- [ ] 是否做了负向验证。
- [ ] 是否排除了管理员正常能力。
- [ ] 是否排除了测试接口、mock 接口、开发接口误判。
- [ ] 是否排除了前端展示问题，而不是后端业务漏洞。
- [ ] 是否检查了幂等、重放、重复提交。
- [ ] 是否检查了异常状态跳转。
- [ ] 是否检查了旧 token、旧 session、旧 API key。
- [ ] 是否检查了 webhook、queue、worker、cron、import/export。
- [ ] 是否存在可复现步骤。
- [ ] 是否不存在推断证据。
- [ ] 是否没有把 candidate 伪装成 confirmed。
- [ ] 是否没有把 blocked 路径误判成漏洞。
- [ ] 是否没有漏掉更严重的同根因变体。

## 2. 第一轮强制反思

- [ ] 是否只看了 HTTP 接口，漏掉 worker、cron、queue、webhook、CLI。
- [ ] 是否只看了前端流程，漏掉后端接受隐藏参数。
- [ ] 是否只看了单用户，漏掉多角色矩阵。
- [ ] 是否只看了单租户，漏掉跨租户业务影响。
- [ ] 是否只看了正常状态，漏掉非法状态跳转。
- [ ] 是否只看了同步请求，漏掉异步补偿和失败重试。
- [ ] 是否只看了单次提交，漏掉重复提交和重放。
- [ ] 是否只看了常规金额，漏掉负数、小数、边界值、精度问题。
- [ ] 是否漏掉旧 token、旧 session、旧 API key。
- [ ] 是否漏掉被禁用、被删除、被移除用户。
- [ ] 是否漏掉管理员降权后的权限残留。
- [ ] 是否漏掉缓存、队列、搜索索引、报表、导出里的租户隔离。
- [ ] 是否把没有数据库变化的响应误判为漏洞。
- [ ] 是否把 401/403 的失败路径误判为成功。
- [ ] 是否有任何 confirmed 没有动态证据。

## 3. 第二轮偏门补挖

- [ ] 是否从数据库字段反向找谁能写、谁会消费。
- [ ] 是否从所有 status 字段反向找状态机缺口。
- [ ] 是否从所有 price、amount、balance、points、quota 字段反向找客户端可控路径。
- [ ] 是否从所有 tenant_id、org_id、owner_id 字段反向找跨租户路径。
- [ ] 是否从所有 webhook handler 反向找幂等、签名、金额校验。
- [ ] 是否从所有 worker job 反向找低权限用户能否投递高权限任务。
- [ ] 是否从所有 admin 接口反向找普通用户是否能间接触发。
- [ ] 是否从所有 invite/reset/verify token 反向找重复使用和过期绕过。
- [ ] 是否从所有 import/export/report 功能反向找越权和状态污染。
- [ ] 是否从所有 cache key 反向找跨用户、跨租户污染。
- [ ] 是否从所有 feature flag 反向找隐藏权限。
- [ ] 是否从所有 plan/subscription 反向找低价高权。
- [ ] 是否从所有审批流反向找自批、越级批、重复批。
- [ ] 是否从所有取消、退款、回滚流程反向找权益残留。
- [ ] 是否从所有异常处理和 retry 逻辑反向找重复业务效果。

## 4. 最终输出修正

- [ ] `confirmed` 只保留动态证据完整、可本地复现的问题。
- [ ] `candidate` 说明为什么可疑、缺少什么证据、如何最小化验证、为什么不能升级 `confirmed`。
- [ ] `blocked` 说明被什么机制阻断：权限、状态机、事务、唯一约束、幂等、租户隔离、后端 allowlist。
- [ ] `false_positive` 说明为什么不成立。
- [ ] `needs_environment` 说明缺少什么环境或数据。
- [ ] 高危漏洞都有负向验证。
- [ ] 不存在编造执行结果。
- [ ] 不存在用推测词替代证据。
- [ ] 环境无法运行时已先修运行环境；无法修复的结论已降级。
