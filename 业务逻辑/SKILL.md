# Business Logic Verify

## 适用范围

本 Skill 只用于“当前本地授权开源项目”的业务逻辑漏洞动态验证。目标必须是本地项目、本地测试环境、测试数据库、测试账号、测试租户、测试订单、测试资金、测试积分、测试优惠券、测试邀请码、测试 webhook 或等价测试数据。

任务目标是基于真实代码、项目结构、依赖、编程语言、框架、数据库结构、权限模型、状态机、业务流程和运行环境，系统性发现并动态验证高影响业务逻辑漏洞。输出结论必须能追溯到本地证据；没有动态证据的可疑点只能进入 `candidate`、`blocked`、`false_positive` 或 `needs_environment`。

## 不适用范围

遇到下列任一情况，停止执行验证并输出失败原因：

- 目标不是当前本地授权项目。
- 环境不是本地测试环境，或涉及生产环境、真实用户、真实订单、真实资金、真实积分、真实优惠券、真实 webhook。
- 用户要求中间人攻击、网络劫持、证书绕过、流量篡改。
- 用户要求 DoS、DDoS、资源耗尽、批量压测、恶意刷接口。
- 用户要求访问公网敏感地址、内网敏感地址、云元数据地址。
- 用户要求绕过授权边界攻击第三方系统。
- 用户要求删除真实数据、破坏数据库、影响真实业务。
- 用户要求在只有静态可疑点时标记 `confirmed`。
- 用户要求编造请求、响应、日志、截图、数据库记录或执行结果。

## 输入要求

执行前建立输入登记表。缺失项不得补写为事实；缺失会影响结论分级。

| 输入项 | 必填规则 | 验收标准 | 缺失处理 |
|---|---|---|---|
| 本地项目路径 | 必填 | 路径存在；能读取源码、依赖文件、配置文件 | `needs_environment`；不执行动态验证 |
| 授权边界 | 必填 | 明确限定本地授权项目和测试环境 | 拒绝验证 |
| 运行方式 | 必填 | 给出启动命令、端口、环境变量、容器或本地服务方式 | 先修本地运行；失败则 `needs_environment` |
| 测试数据库或快照 | 必填 | 测试库、fixture、事务回滚或快照可用 | 不做写入型验证；相关结论不得 `confirmed` |
| 测试账号和角色 | 必填 | 覆盖低权限、高权限、被禁用、被移除、降权后或项目等价角色 | 缺失角色矩阵则权限类结论不得 `confirmed` |
| 测试租户 | 多租户项目必填 | 至少 tenant A 和 tenant B；每个租户至少两个用户 | 跨租户结论不得 `confirmed` |
| 测试业务对象 | 按项目存在项必填 | 订单、支付、退款、优惠券、余额、积分、会员、订阅、邀请、文件、项目等 | 不得编造对象；对应项降级 |
| webhook/mock 回调 | 项目存在 webhook 时必填 | 本地 mock webhook 可构造签名、timestamp、event_id | 不调用真实第三方；缺失则 webhook 结论不得 `confirmed` |
| 日志采集方式 | 必填 | 能读取后端日志、worker 日志、队列日志或等价本地证据 | 缺少日志时不得 `confirmed` |
| DB 查询方式 | 必填 | 能记录触发前后数据库状态 | 缺少 DB 前后状态时不得 `confirmed` |
| 业务规则说明 | 推荐 | 提供订单、支付、退款、审批、邀请、订阅、会员等规则 | 未提供时从代码和数据结构反推，并标注来源 |

## 输出要求

最终产物是“业务逻辑漏洞动态验证报告”，必须包含：

1. 项目业务暴露面总览。
2. 项目理解结果。
3. replay plan 清单。
4. 已确认业务逻辑漏洞。
5. 候选漏洞。
6. 被阻断路径。
7. false_positive。
8. needs_environment。
9. 工程级修复建议。
10. 第一轮强制反思。
11. 第二轮偏门补挖。
12. 每个 `confirmed` 的最终审判和修正后结论。
13. 证据索引。

每个 `confirmed` 必须包含：标题、严重性、影响范围、涉及角色、涉及租户、涉及业务对象、被破坏的业务不变量、正常流程、异常绕过流程、动态复现步骤、请求和响应、数据库前后变化、日志证据、相关代码位置、根因分析、修复建议、修复后回归测试、至少一个负向验证样本。

## 原文复刻规则

本节为 TXT 原文要求的可执行化复刻。执行时不得弱化边界、不得把静态推断写成动态事实、不得把工程化补强写成原文要求。

### 1. 强制边界

- 只允许分析和验证当前本地授权项目。
- 只允许使用本地测试环境、测试数据库、测试账号、测试租户、测试订单、测试资金、测试积分、测试优惠券、测试邀请码、测试 webhook。
- 禁止破坏数据库、删除真实数据、影响真实业务。
- 禁止 DoS、DDoS、资源耗尽、批量压测、恶意刷接口。
- 禁止访问公网敏感地址、内网敏感地址、云元数据地址。
- 禁止绕过授权边界攻击第三方系统。
- 所有验证必须可回滚，必须使用测试数据，必须保留证据。
- 只有静态可疑点、没有动态复现证据时，只能标记 `candidate`，不得标记 `confirmed`。
- 不允许编造请求、响应、日志、截图、数据库记录或执行结果。

### 2. 项目理解阶段

先完整理解项目，不急于下结论。必须识别并记录：

- 语言、框架、入口文件、路由、控制器、service、model、middleware、worker、队列、定时任务。
- 认证方式：session、JWT、OAuth、API key、magic link、邮箱验证码、手机验证码、webhook signature、admin token、service token。
- 权限模型：匿名用户、普通用户、管理员、超级管理员、组织 owner、组织 member、审批人、商家、客服、代理、多租户用户、内部服务账号；只记录项目实际存在项。
- 核心业务对象：用户、组织、租户、订单、支付、退款、优惠券、余额、积分、会员、订阅、发票、邀请、审批、工单、文件、项目、Webhook、API token、邮箱绑定、OAuth 绑定、密码重置；只记录项目实际存在项。
- 关键状态机：订单、支付、退款、审批、邀请、订阅、会员、发货、工单、邮箱验证、账号绑定。
- 业务流程图：正常流程、异常流程、取消流程、重试流程、回滚流程、管理员干预流程、webhook 异步回调流程、队列补偿流程。

### 3. 十类业务逻辑漏洞方向

对项目实际存在的功能逐类建立候选点和 replay plan：

| 类别 | 必查点 | 动态验证硬规则 |
|---|---|---|
| 权限错位 | 普通用户调管理员接口、member 执行 owner 操作、A 用户操作 B 用户资源、A 租户访问 B 租户资源、禁用/移除/降权后权限残留、后端接受 role/is_admin/tenant_id/owner_id/status、只靠前端隐藏按钮 | 至少两个用户、两个角色、两个租户；低权限尝试高权限动作；跨租户尝试；记录请求、响应、DB、日志 |
| 状态机绕过 | 未支付发货、已取消支付、已退款重复退款、已完成审批再次修改、过期邀请使用、未验证邮箱绑定、跳过前置步骤、客户端提交 status、状态回退恢复权益、重试造成错乱 | 构造正常状态流和非法跳转；检查 DB；验证后端拒绝；不得只看前端 |
| 金额、余额、积分、优惠 | 负数金额、小数精度、四舍五入、优惠叠加、重复使用、跨用户/跨租户、过期券、前端满减、余额与订单不一致、退款后未回滚、取消后权益残留、低价高权、试用重复、会员到期权益残留 | 只用测试金额；不调用真实支付；使用 mock 支付或本地回调；对比订单、余额、积分、优惠券、权益表；证明业务结果变化 |
| 幂等和重复提交 | 支付/退款/订单/邀请/优惠券/积分/审批/验证码/重置 token/webhook 重复处理 | 相同请求重复发送；相同 idempotency key；缺失 key；模拟 webhook 重放；检查重复写入、唯一约束、事务、幂等表、状态锁 |
| 并发竞争 | 库存超卖、余额重复扣减或少扣、优惠券超领、邀请码超用、审批重复通过、并发退款、并发绑定、token 并发消费、租户配额突破、免费额度突破 | 仅 2 到 5 个并发请求；不压测；记录每个响应和最终 DB；判断是否违反业务不变量 |
| Webhook 和异步回调 | 签名缺失、只验格式、timestamp 不校验、event_id 不幂等、状态伪造、金额不一致仍成功、user_id/tenant_id/order_id 篡改、worker 高权限处理、失败重试重复发权益 | 使用本地 mock webhook；不攻击真实第三方；构造合法签名、非法签名、过期 timestamp、重复 event_id；对比业务结果、worker 日志、DB |
| 账号生命周期 | 未验证邮箱使用高权限、邮箱绑定未验证、OAuth 错绑、重置 token 复用、重置后旧 session/token 有效、改邮箱后旧邮箱登录、删号/禁用后 token 或 API key 有效、移除成员仍访问、邀请过期/撤销/复用 | 建完整生命周期测试；使用旧 token/session/API key 复测；检查 DB 状态与实际权限一致性 |
| 后端接受隐藏参数 | role、is_admin、price、discount、quota、plan、status、owner_id、tenant_id、user_id、approved、verified、balance、points、permission、feature_flags、internal、callback_url、redirect_url、webhook_url | 从正常前端请求提取参数；增加隐藏参数；观察后端是否采纳；对比 DB；不能只看响应文案 |
| 多租户隔离 | 客户端提交 tenant_id、查询只按 id、worker 丢 tenant context、webhook 只按 order_id、导入导出/报表/邀请/优惠券/后台跨租户、cache key 无 tenant_id、queue topic 未隔离 | tenant A/B；每租户至少两个用户；创建订单、优惠券、邀请、文件、项目；跨租户读写审批退款导出；检查缓存、队列、worker、webhook tenant context |
| 业务规则反推 | 从核心业务不变量反推漏洞：未支付不能发货、已退款不能再退、优惠券一次、余额不能负、成员不能自批、移除成员不能访问、跨租户隔离、试用一次、webhook 金额匹配、重置 token 一次、邀请未过期未撤销未超次数 | 对每条不变量设计动态验证；输出正常路径、非法路径、证据、结论 |

### 4. 动态验证标准与 replay plan

每个候选漏洞必须生成 replay plan，字段不可缺省：

1. 漏洞编号。
2. 业务对象。
3. 正常业务流程。
4. 被测试的不变量。
5. 攻击前提：用户角色、租户、数据状态、token/session、业务对象 id。
6. 触发入口：HTTP API、GraphQL、WebSocket、CLI、webhook、worker、admin panel、import/export、background job。
7. 请求构造方式。
8. 正向验证：构造违反业务规则的请求，观察后端是否接受，检查数据库、日志、状态变化、权益变化。
9. 负向验证：合法用户合法流程应成功；非法用户非法流程应失败；跨租户操作应失败；重放请求应失败；修复后应失败。
10. 证据：请求、响应、数据库前后快照、关键日志、相关代码位置、业务状态变化、角色和租户矩阵、前端截图或 trace。
11. 结论：`confirmed`、`candidate`、`blocked`、`false_positive`、`needs_environment`。

### 5. 证据要求

每个 `confirmed` 业务逻辑漏洞至少具备：

- 违反的业务不变量。
- 触发请求。
- 触发前数据库状态。
- 触发后数据库状态。
- 后端响应。
- 关键日志。
- 相关代码位置。
- 低权限或异常状态下成功完成本不应成功的业务动作。
- 至少一个负向验证样本。

禁止误判：只有代码片段、只有 success 文案、只有前端异常、只有理论推断、环境配置问题、管理员正常能力、测试/mock/开发接口不可达生产路径、失败响应、401/403 均不得写成 `confirmed`。

### 6. 输出格式

最终报告必须包含五个主段落，并加上反思、补挖和最终审判：

1. 项目业务暴露面总览：模块、业务对象、入口、角色、租户边界、关键状态、关键不变量、是否动态验证、风险等级、当前结论。
2. 已确认业务逻辑漏洞：标题、严重性、影响范围、角色、租户、对象、不变量、正常流程、绕过流程、复现步骤、请求响应、DB 前后、日志、代码、根因、修复、回归测试。
3. 候选漏洞：为什么可疑、缺少什么证据、如何最小化动态验证、为什么不能升级 `confirmed`。
4. 被阻断路径：被权限、状态机、schema、唯一约束、事务、幂等、租户隔离、后端 allowlist 中哪一种机制阻断。
5. 修复建议：后端权限、租户隔离、服务端状态机、禁止客户端敏感字段、金额积分余额服务端计算、webhook 签名/timestamp/event_id、事务、幂等 key、唯一约束、token 一次性消费、账号状态变更撤销旧凭证、worker tenant context、缓存 key tenant_id、审计日志、回归测试。

### 7. 第一轮强制反思

第一轮报告完成后不得结束。必须反查是否漏掉：HTTP 以外的 worker/cron/queue/webhook/CLI、隐藏参数、多角色矩阵、多租户、非法状态跳转、异步补偿和失败重试、重复提交和重放、负数/小数/边界值/精度、旧 token/session/API key、禁用/删除/移除用户、管理员降权残留、缓存/队列/搜索索引/报表/导出租户隔离、无 DB 变化的 success、401/403 被误判、任何无动态证据的 `confirmed`。

### 8. 第二轮偏门补挖

第二轮从非常规入口补挖：数据库字段写入者和消费者、所有 status、price/amount/balance/points/quota、tenant_id/org_id/owner_id、webhook handler、worker job、admin 接口、invite/reset/verify token、import/export/report、cache key、feature flag、plan/subscription、审批流、取消/退款/回滚、异常处理和 retry。

### 9. 最终硬性要求

最终结论必须重判：`confirmed` 只能是动态证据完整、可本地复现；`candidate` 不能伪装成 `confirmed`；`blocked` 要说明阻断机制；`false_positive` 要说明不成立原因；`needs_environment` 要说明缺少环境或数据；每个高危漏洞必须有负向验证；不得编造执行结果；不得用推测词替代证据；环境无法运行时先修运行环境，无法修复则降级。

### 10. confirmed 反向审判

逐个审查每个 `confirmed`：不变量、动态请求响应、DB 前后、日志/worker、业务结果、低权限或异常状态、跨租户影响、负向验证、排除管理员正常能力、排除测试/mock/开发接口误判、排除前端展示问题、幂等/重放/重复提交、异常状态、旧 token/session/API key、webhook/queue/worker/cron/import/export、可复现步骤、无推断证据、未把 candidate 伪装、未把 blocked 误判、未漏同根因严重变体。

## 工程化补强规则

以下内容为工程化补强，用于让 Skill 可调用、可填写、可验收，不得声称是 TXT 原文逐字要求：

- 使用证据编号：`EV-REQ-*`、`EV-RESP-*`、`EV-DB-BEFORE-*`、`EV-DB-AFTER-*`、`EV-LOG-*`、`EV-CODE-*`、`EV-TRACE-*`。
- 使用候选编号：`BLV-YYYYMMDD-NNN`。
- 每个报告表格必须保留“证据编号”列。
- 使用 `needs_environment` 表示环境或测试数据不足；该分级来自 TXT 末尾复核要求中的 `needs_environment`。
- 对敏感 token、cookie、key、用户隐私字段做脱敏记录；保留能复现权限矩阵的标识。
- 先输出 replay plan，再执行动态验证；执行失败时记录失败命令、错误、日志位置和降级理由。

## 核心工作流

1. 范围确认：确认本地授权、测试数据、可回滚、禁止项。
2. 项目画像：识别语言、框架、入口、路由、控制器、service、model、middleware、worker、queue、cron。
3. 权限与租户画像：认证方式、角色矩阵、租户矩阵、旧 token/session/API key。
4. 业务对象和不变量：对象、状态机、正常/异常/取消/重试/回滚/管理员干预/webhook/队列补偿流程。
5. 十类方向建模：为项目实际存在项建立候选点。
6. replay plan：为每个候选点填写 11 项字段。
7. 动态验证：只在本地测试环境执行，记录请求、响应、DB 前后、日志、代码位置、截图/trace。
8. 初始分级：confirmed/candidate/blocked/false_positive/needs_environment。
9. 第一轮强制反思：按 TXT 15 项反查并修正。
10. 第二轮偏门补挖：按 TXT 15 项反向入口补挖。
11. confirmed 审判：按 TXT 20 项逐个审查；证据不足立即降级。
12. 输出最终报告和证据索引。

## 分阶段执行步骤

### 阶段 0：安全边界确认

输入：用户提供的项目路径、授权边界、测试环境说明。输出：范围确认表。判断标准：任何禁止项触发即停止验证。失败处理：输出拒绝原因和需要补充的本地测试条件。

### 阶段 1：项目画像

输入：源码目录、依赖文件、配置文件、路由文件、DB schema、任务队列配置。输出：项目理解表。判断标准：必须覆盖同步入口和异步入口。失败处理：无法读取的组件标记 `needs_environment`，不得编造。

### 阶段 2：认证、权限、租户画像

输入：认证配置、中间件、角色表、租户表、测试账号。输出：角色/租户矩阵。判断标准：权限类验证至少包含低权限、高权限、跨用户、跨租户路径。失败处理：缺少账号或租户时对应结论不得 `confirmed`。

### 阶段 3：业务对象、状态机和不变量

输入：DB schema、model、service、控制器、前端流程、队列任务。输出：业务对象表、状态机表、不变量清单。判断标准：每个核心对象至少有合法状态流和非法状态跳转候选。失败处理：对象不存在则记录“不适用”，不得硬套。

### 阶段 4：生成 replay plan

输入：候选点、不变量、角色/租户矩阵、测试数据。输出：replay plan 表。判断标准：11 项字段全部填写。失败处理：缺字段的 plan 不得执行为 confirmed 验证。

### 阶段 5：动态验证和证据采集

输入：replay plan、本地服务、测试 DB、日志。输出：请求、响应、DB 前后、日志、代码位置、业务状态变化。判断标准：业务结果发生并破坏不变量时才允许进入 `confirmed` 候选。失败处理：环境错误进入 `needs_environment`；被阻断进入 `blocked`；只有静态证据进入 `candidate`。

### 阶段 6：分级和报告

输入：证据集合。输出：初版报告。判断标准：`confirmed` 必须满足证据门禁；`candidate/blocked/false_positive/needs_environment` 必须说明原因。失败处理：证据不足立即降级。

### 阶段 7：第一轮强制反思

输入：初版报告。输出：修正记录。判断标准：TXT 15 项反思全部填写。失败处理：发现漏项则补查；无法补查则对应结论降级。

### 阶段 8：第二轮偏门补挖

输入：DB 字段、状态字段、金额字段、租户字段、webhook、worker、admin、token、导入导出、cache、feature flag、订阅、审批、回滚、retry。输出：新增候选点和补充 replay plan。判断标准：每个新增候选点必须分级。失败处理：不能验证的新增点进入 `candidate` 或 `needs_environment`。

### 阶段 9：最终 confirmed 审判

输入：所有 `confirmed`。输出：最终重判表。判断标准：20 项 confirmed 审判全部通过才保留 `confirmed`。失败处理：任何关键证据缺失即降级。

## 质量门禁

- 未确认本地授权和测试环境，不得执行验证。
- 未读取源码、依赖、入口、路由、DB schema、异步任务，不得生成完整报告。
- 未建立角色矩阵和租户矩阵，权限或跨租户结论不得 `confirmed`。
- 未建立 TXT 到报告字段的映射，不得交付。
- 未区分原文复刻和工程化补强，不得交付。
- 未覆盖强制边界、项目理解、十类漏洞、replay plan、证据要求、输出格式、第一轮反思、第二轮补挖、最终审判，不得交付。
- 未提供请求、响应、DB 前后、日志、代码位置、负向验证，不得标记 `confirmed`。
- 未记录 blocked 的阻断机制，不得写 blocked。
- 未说明 false_positive 不成立原因，不得写 false_positive。
- 未说明 needs_environment 缺失条件，不得写 needs_environment。
- 未执行第一轮反思和第二轮补挖，不得输出最终结论。
- 任一证据为推断或占位，不得写 `confirmed`。

## 幻觉控制

- 不把源码中不存在的业务对象写成已验证对象。
- 不把未运行的请求写成已执行请求。
- 不把静态代码位置写成动态复现。
- 不把前端 success 文案写成业务状态变化。
- 不把 401、403、409 等失败响应写成漏洞成功。
- 不把管理员正常操作写成越权。
- 不把测试、mock、开发接口写成真实运行路径，除非证明该路径在本地目标运行环境可达且属于验证范围。
- 不把环境配置错误写成业务漏洞。
- 不把工程化补强写成 TXT 原文。

## 失败处理

| 失败场景 | 输出结论 | 处理规则 |
|---|---|---|
| 未授权或非本地目标 | 拒绝执行 | 说明违反边界，不给验证步骤 |
| 无测试数据库或不可回滚 | `needs_environment` | 不执行写入型验证 |
| 服务无法启动 | `needs_environment` | 记录启动命令、错误、日志；不得伪造动态结果 |
| 缺少角色或租户 | `needs_environment` 或 `candidate` | 对权限/跨租户方向降级 |
| 只有代码可疑点 | `candidate` | 写明缺少请求、响应、DB、日志中的哪些证据 |
| 请求被权限/状态/事务/唯一约束/幂等/租户隔离阻断 | `blocked` | 写明阻断机制和证据 |
| 误读管理员能力、测试路径、前端展示或失败响应 | `false_positive` | 写明不成立原因 |
| confirmed 缺任一硬证据 | 降级 | 降级到 candidate、blocked、false_positive 或 needs_environment |

## 输出格式

使用 `templates/output-template.md`。不得删除模板中的分级、证据编号、第一轮反思、第二轮补挖、最终审判、证据索引。项目没有某类对象时，写“项目未发现该对象/入口，证据：<文件或搜索结果>”，不得空着。
