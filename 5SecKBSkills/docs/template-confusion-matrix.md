# 模板混淆矩阵

本矩阵用于阻止 Claude 因关键词相似而误调用漏洞模板。每次从候选模板进入验证前，必须先检查相似模板的区分条件。

## SSRF vs URL fetch

错误模式：看到 `fetch`、`http.Client`、`requests.get`、`axios` 就直接报告 SSRF。

区分条件：

- SSRF 必须存在可控 URL 到服务端请求的链路。
- 必须能在本机或授权环境中证明服务端发起请求。
- 如果 URL 固定、白名单严格、仅客户端请求、不可控，则不能按 SSRF 报告。

## RCE vs 命令执行危险函数

错误模式：看到 `exec`、`spawn`、`Runtime.exec` 就直接报告 RCE。

区分条件：

- RCE 必须存在外部输入影响命令或参数的完整链路。
- 仅危险函数存在不是漏洞。
- 测试代码、构建脚本、固定命令、参数不可控时不能报告 RCE。

## 信息泄露 vs 可报告敏感数据暴露

错误模式：把版本号、README、错误页面、示例密钥直接写成高危。

区分条件：

- 可报告敏感数据暴露必须有实际敏感内容或可证明安全影响。
- 证明影响时必须脱敏，不能读取真实敏感数据。
- 只有版本号、公开 README、示例数据、普通报错通常不能直接报告。

## IDOR vs 正常权限差异

错误模式：把正常管理员权限、UI 差异、角色差异写成越权。

区分条件：

- IDOR/BOLA 必须证明跨用户、跨租户或跨对象访问了不属于自己的资源。
- 必须有两个账户、对象归属证据和服务端授权缺陷证据。
- 管理员正常可见、业务设计允许、无对象归属证据时不能报告 IDOR。
