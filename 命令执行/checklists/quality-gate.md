# quality-gate

本清单是执行 `cmd-exec-risk-replay` 时的硬门禁。任一必需项失败，必须停止、降级或回到前一阶段补证据。

## 1. 边界门禁

- [ ] 已确认项目路径属于当前本地授权项目。
- [ ] 已确认授权范围，不含未授权目标。
- [ ] 已确认只使用本地测试环境、测试数据库、测试目录。
- [ ] 已确认不访问公网敏感地址、内网敏感地址、云元数据地址。
- [ ] 已确认不读取真实密钥、真实用户数据、真实配置秘密。
- [ ] 已确认不执行反弹 shell、下载执行、持久化、提权、横向移动、凭据读取、删库、删文件、业务破坏、DoS、DDoS、资源耗尽。
- [ ] 已确认本轮不处理 MITM、网络劫持、证书绕过。
- [ ] 已确认 marker 只写入项目测试目录，例如 `./security-lab/markers/`。

## 2. 输入门禁

- [ ] 已记录 `project_root`。
- [ ] 已记录 `authorization_scope`。
- [ ] 已记录操作系统。
- [ ] 已记录测试环境启动方式或无法启动原因。
- [ ] 已记录测试数据库或测试目录。
- [ ] 已记录 marker 目录和证据目录。
- [ ] 已记录可用角色和租户；没有角色/租户时写明影响。

## 3. 项目理解门禁

- [ ] 已识别语言和框架。
- [ ] 已识别入口文件和启动方式。
- [ ] 已识别路由注册方式。
- [ ] 已识别认证方式和权限模型。
- [ ] 已检查 CLI。
- [ ] 已检查 worker、queue、cron、scheduler、job。
- [ ] 已检查 webhook processor。
- [ ] 已检查导入、导出、备份、恢复、报表、预览、渲染、文件转换。
- [ ] 已检查插件、hook、extension、event listener、ORM hook、生命周期 hook。
- [ ] 未发现的项目面已写“未发现”；无法验证的项目面已写“未验证+原因”。

## 4. source 门禁

- [ ] 已检查 HTTP 参数。
- [ ] 已检查 JSON body。
- [ ] 已检查 multipart 文件名、字段名、content-type。
- [ ] 已检查 header、cookie。
- [ ] 已检查 WebSocket 消息。
- [ ] 已检查 GraphQL 参数。
- [ ] 已检查 webhook body。
- [ ] 已检查 CLI 参数。
- [ ] 已检查环境变量和配置文件。
- [ ] 已检查数据库中可被用户写入的字段。
- [ ] 已检查队列消息。
- [ ] 已检查模板变量、文件路径、压缩包条目名。
- [ ] 已检查插件配置和第三方集成参数。

## 5. sink 门禁

- [ ] 已检查 shell 命令执行 API。
- [ ] 已检查子进程 API。
- [ ] 已检查任务调度 API。
- [ ] 已检查脚本执行 API。
- [ ] 已检查模板渲染到命令链。
- [ ] 已检查文件转换工具调用。
- [ ] 已检查压缩、解压、图片、视频、PDF、Git、包管理器、构建工具、数据库客户端、系统工具包装器。
- [ ] 已检查自定义 wrapper、helper、adapter、service、plugin、hook。
- [ ] 没有只停留在 grep API 名称；已追踪别名、间接调用、异步链和回调。

## 6. 数据流门禁

- [ ] 每条候选均有 source。
- [ ] 每条候选均有 transform 或传递节点。
- [ ] 每条候选均有 sink。
- [ ] 每条候选均有代码位置或配置位置。
- [ ] 每条候选均标注权限前提。
- [ ] 每条候选均标注租户影响或“不适用”。
- [ ] 已记录 allowlist、escaping、schema validation、权限校验、参数数组、固定工作目录、固定二进制路径。

## 7. replay plan 门禁

- [ ] 每个候选均有漏洞编号。
- [ ] 每个候选均有入口点。
- [ ] 每个候选均有权限前提。
- [ ] 每个候选均有 source → transform → sink。
- [ ] 每个候选均有本地触发步骤。
- [ ] 每个候选均有唯一无害 marker。
- [ ] 每个候选均有 marker 允许位置。
- [ ] 每个候选均有正向验证设计。
- [ ] 每个候选均有负向验证设计。
- [ ] 每个候选均有证据字段。
- [ ] 没有 replay plan 的候选不得升级为 `confirmed`。

## 8. 动态验证门禁

- [ ] 正向验证只产生无害 marker。
- [ ] 负向验证覆盖正常输入。
- [ ] 负向验证覆盖转义后输入或等价安全样本。
- [ ] 负向验证覆盖权限不足角色；项目无该角色时写明原因。
- [ ] 负向验证覆盖其他租户；项目无租户时写明原因。
- [ ] 修复后验证方法已给出。
- [ ] marker 不是普通字符串回显。
- [ ] marker 不是只出现在日志中。
- [ ] 证据能证明数据进入真实执行 sink。

## 9. 结论门禁

- [ ] `confirmed` 有动态验证成功证据。
- [ ] `confirmed` 至少有两类交叉证据。
- [ ] `confirmed` 有负向验证。
- [ ] `confirmed` 有 source 到 sink 真实可达证明。
- [ ] 静态可疑但动态未确认的项为 `candidate`。
- [ ] 被权限、校验、allowlist、参数数组、固定工作目录、不可达逻辑阻断的项为 `blocked`。
- [ ] 数据无法到达 sink 或 marker 只是普通回显的项为 `false_positive`。
- [ ] 环境不足或依赖不可运行的项为 `needs_environment`。
- [ ] 依赖 CVE 未证明当前项目可达时，不得写 `confirmed`。
- [ ] 测试/demo/开发脚本未证明运行时可触发时，不得写 `confirmed`。

## 10. 输出门禁

- [ ] 报告包含项目命令执行暴露面总览。
- [ ] 报告包含已确认漏洞。
- [ ] 报告包含候选漏洞。
- [ ] 报告包含被阻断路径。
- [ ] 报告包含 false positive。
- [ ] 报告包含 needs_environment。
- [ ] 报告包含工程级修复项。
- [ ] 报告包含第二轮反思。
- [ ] 报告包含第三轮反常规补挖。
- [ ] 报告包含 confirmed 反向审判。
- [ ] 报告明确哪些验证没有完成及原因。
