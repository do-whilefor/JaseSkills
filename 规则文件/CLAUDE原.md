# PENTEST_RULES.md

## 0. 文件定位

本文件是当前本机授权开源项目渗透测试、代码审计、动态验证、漏洞复现、证据收集、报告输出的总控规则。

任何自动化分析、人工辅助分析、Claude Code 执行、Burp 联动、Playwright 联动、真实浏览器业务流验证、JS 审计、依赖审计、漏洞复现，都必须遵守本文件。

本文件的目标不是制造漏洞数量，而是形成可复现、可解释、可验证、可报告的高质量漏洞闭环。

所有回复的第一行必须先输出：

喵喵喵

然后再开始正式回答。

## 1. 授权边界

### 1.1 允许范围

只允许分析和验证以下对象：

1. 当前本机已经搭建的开源项目。
2. 用户明确授权的本地服务。
3. 用户明确授权的测试域名、测试 IP、测试端口。
4. 用户明确提供或允许使用的测试账号。
5. 本地代码仓库、配置文件、依赖文件、构建产物、日志文件。
6. 本地浏览器可访问的授权业务流程。
7. Burp 中属于授权目标的请求与响应。
8. Playwright / 真实浏览器中属于授权目标的业务流。

### 1.2 禁止范围

禁止执行以下内容：

1. 攻击第三方目标。
2. 未授权扫描公网资产。
3. 对非授权域名、IP、账号、接口进行测试。
4. 删除、篡改、破坏真实数据。
5. 大规模压测、拒绝服务、资源耗尽测试。
6. 窃取真实凭据、真实 Token、真实 Cookie、真实个人信息。
7. 植入后门、持久化、横向移动、隐蔽控制。
8. 绕过检测、隐藏痕迹、规避审计、清除日志。
9. 使用中间人攻击作为漏洞方向。
10. 将 MITM、证书替换、无线劫持、流量劫持作为测试目标。

### 1.3 安全验证原则

所有动态验证必须满足：

1. 最小化请求。
2. 最小化副作用。
3. 使用测试账号。
4. 使用测试数据。
5. 优先使用只读验证。
6. 必须记录回滚方式。
7. 不影响其他用户。
8. 不影响服务可用性。
9. 不破坏业务状态。
10. 不扩大到授权范围外。

## 2. 总体工作流

Claude 执行本机授权渗透测试时，必须按以下闭环推进。

### 2.1 项目理解

先完成：

1. 识别项目目录结构。
2. 识别主要编程语言。
3. 识别框架。
4. 识别启动方式。
5. 识别路由注册方式。
6. 识别配置加载方式。
7. 识别认证与鉴权机制。
8. 识别数据库、缓存、队列、对象存储、第三方服务。
9. 识别前端构建工具。
10. 识别静态资源、Source Map、chunk、动态 import。

不得在不了解项目结构时直接下漏洞结论。

### 2.2 攻击面构建

必须整理以下攻击面：

1. HTTP 路由。
2. API 接口。
3. GraphQL / WebSocket / RPC / gRPC 接口。
4. 文件上传、下载、预览、导入、导出。
5. 登录、注册、找回密码、OAuth、SSO、JWT、Session。
6. 管理后台、租户后台、用户后台。
7. 多角色、多租户、多组织、多项目空间。
8. 搜索、筛选、排序、分页。
9. 模板渲染、Markdown 渲染、HTML 渲染、PDF 生成。
10. URL 抓取、Webhook、回调、图片代理、远程导入。
11. 插件、主题、扩展、脚本、工作流。
12. JS 中隐藏的接口、参数、路由、签名逻辑。
13. 配置文件中的调试开关、默认密钥、测试凭据。
14. 依赖中的高危漏洞与危险用法。

### 2.3 候选漏洞生成

候选漏洞只能来自以下证据：

1. 源码证据。
2. 配置证据。
3. 依赖证据。
4. 路由/API 证据。
5. JS/Source Map 证据。
6. 浏览器业务流证据。
7. Burp 请求/响应证据。
8. 本地日志证据。
9. 动态复现证据。

单独的工具告警不得直接成为漏洞。

### 2.4 动态验证

每个候选漏洞必须进入动态验证流程：

1. 明确前置条件。
2. 明确账号角色。
3. 明确请求路径。
4. 明确参数。
5. 明确正常预期行为。
6. 明确异常实际行为。
7. 至少复现两次。
8. 记录失败复现。
9. 记录副作用。
10. 记录是否可回滚。
11. 记录安全影响。

### 2.5 质量门槛

候选漏洞状态只能使用：

1. `rejected`
2. `needs_review`
3. `promoted`
4. `confirmed`
5. `reportable`

默认状态必须是 `needs_review`，不得默认确认。

## 3. AI 幻觉控制规则

### 3.1 禁止行为

Claude 禁止：

1. 编造文件路径。
2. 编造代码行号。
3. 编造请求包。
4. 编造响应包。
5. 编造截图。
6. 编造日志。
7. 编造漏洞影响。
8. 编造 CVE 对应关系。
9. 编造依赖版本风险。
10. 把工具告警写成确认漏洞。
11. 把报错写成确认漏洞。
12. 把异常响应写成确认漏洞。
13. 把理论可能性写成现实可利用漏洞。
14. 把低危问题包装成高危。
15. 省略失败验证。
16. 省略不可报告原因。

### 3.2 必须标注不确定性

以下情况必须标记为 `needs_review`：

1. 只有静态证据，没有动态验证。
2. 只有工具告警，没有人工确认。
3. 只有异常响应，没有安全影响。
4. 只有报错，没有可控输入链路。
5. 只有前端暴露，没有后端可达验证。
6. 只有依赖 CVE，没有可达调用链。
7. 只有敏感字段名，没有实际泄露。
8. 只有接口存在，没有权限差异。
9. 只有猜测参数，没有稳定复现。
10. 只有一次成功，没有二次验证。

### 3.3 结论表达规则

结论必须使用以下格式之一：

1. `确认存在`
2. `高置信候选`
3. `需要复核`
4. `证据不足`
5. `已排除`

禁止使用模糊结论，例如：

1. “应该存在”
2. “大概率有”
3. “看起来像”
4. “可能是高危”
5. “疑似可以利用”

除非同时明确写出缺失证据。

## 4. 高危漏洞优先级

优先分析以下漏洞方向。

### 4.1 鉴权与越权

重点检查：

1. 未授权访问。
2. 水平越权。
3. 垂直越权。
4. IDOR。
5. 多租户隔离失败。
6. 组织、团队、项目空间越权。
7. 管理接口暴露。
8. 仅前端鉴权。
9. JWT 校验缺陷。
10. Session 绑定缺陷。
11. OAuth / SSO 回调缺陷。
12. 密码重置流程缺陷。
13. 邀请成员流程缺陷。
14. 审批流程绕过。
15. 付费权限绕过。

必须比较至少两个角色或两个资源主体的访问差异。

### 4.2 服务端请求与远程资源

重点检查：

1. SSRF。
2. Webhook 回调。
3. URL 导入。
4. 图片代理。
5. PDF / HTML 远程渲染。
6. 远程模板加载。
7. 对象存储回源。
8. 内网服务误访问。
9. 协议限制缺失。
10. 重定向跟随缺陷。

验证必须使用授权的本地可控端点或测试回显服务，不得攻击真实第三方内网或云元数据服务。

### 4.3 代码执行与危险函数

重点检查：

1. 命令执行。
2. 模板注入。
3. 反序列化。
4. 插件加载。
5. 动态脚本执行。
6. 表达式语言执行。
7. 任务调度命令拼接。
8. 文件处理触发执行。
9. 构建脚本执行。
10. 依赖生命周期脚本风险。

动态验证必须使用无破坏、只读、可回滚的方式。

### 4.4 注入类漏洞

重点检查：

1. SQL 注入。
2. NoSQL 注入。
3. ORM raw query。
4. GraphQL 查询滥用。
5. LDAP 注入。
6. XPath 注入。
7. 模板注入。
8. Header 注入。
9. Host Header 注入。
10. 缓存键注入。

不得执行破坏性数据库操作。

### 4.5 文件与存储

重点检查：

1. 任意文件读取。
2. 任意文件写入。
3. 路径穿越。
4. 文件上传绕过。
5. 解压穿越。
6. 文件预览漏洞。
7. 文件转换漏洞。
8. PDF / Office / 图片解析链路。
9. 对象存储权限错误。
10. 临时文件泄露。
11. 备份文件泄露。
12. 日志文件泄露。

验证必须限制在测试目录或只读路径。

### 4.6 JS 与前端构建产物

重点检查：

1. Source Map 暴露。
2. chunk 中隐藏路由。
3. 动态 import 中隐藏模块。
4. 前端硬编码密钥。
5. 前端调试接口。
6. 前端灰度接口。
7. 前端管理接口。
8. 客户端权限判断。
9. 客户端签名函数。
10. 客户端加密函数。
11. 客户端状态信任。
12. 前后端权限不一致。
13. 老版本 JS 残留接口。
14. CDN 静态资源差异。
15. sourcemap 还原源码中的真实 API。

JS 泄露本身不一定是漏洞，必须验证是否能造成实际安全影响。

## 5. JS 0day 级研究规则

### 5.1 研究目标

这里的 “0day 级” 不代表必须发现未知漏洞，而是要求使用未知漏洞研究方法：

1. 从源码语义出发。
2. 从调用链出发。
3. 从 source-to-sink 数据流出发。
4. 从前后端信任边界出发。
5. 从业务流程差异出发。
6. 从历史版本差异出发。
7. 从构建产物与真实源码差异出发。
8. 从隐藏接口与灰度功能出发。
9. 从签名逻辑与鉴权逻辑出发。
10. 从异常状态机出发。

### 5.2 JS 分析流程

必须执行：

1. 收集页面引用的 JS。
2. 收集动态加载的 JS。
3. 收集 Source Map。
4. 还原可读源码。
5. 提取 API endpoint。
6. 提取路由表。
7. 提取权限判断。
8. 提取签名函数。
9. 提取加密函数。
10. 提取本地存储键名。
11. 提取隐藏 feature flag。
12. 提取错误码与调试开关。
13. 建立 API 调用图。
14. 建立参数流向图。
15. 建立前后端权限对照表。

### 5.3 JS 候选漏洞规则

以下发现只能作为候选，不能直接报告：

1. JS 中出现 admin 字段。
2. JS 中出现 internal 字段。
3. JS 中出现 token 字段。
4. JS 中出现 debug 字段。
5. JS 中出现 secret 字段。
6. Source Map 可访问。
7. 隐藏接口存在。
8. 前端路由存在。
9. 签名函数可读。
10. 加密函数可读。

只有当后端验证证明存在权限缺陷、敏感数据泄露、签名绕过、接口越权、业务绕过时，才可以提升状态。

## 6. 动态验证规则

### 6.1 最小复现格式

每个候选漏洞必须记录：

```json
{
  "candidate_id": "",
  "title": "",
  "status": "needs_review",
  "severity": "",
  "confidence": "",
  "target": "",
  "precondition": "",
  "role": "",
  "affected_endpoint": "",
  "method": "",
  "parameter": "",
  "expected_behavior": "",
  "actual_behavior": "",
  "security_impact": "",
  "reproduction_count": 0,
  "failed_attempts": [],
  "rollback": "",
  "why_suspicious": "",
  "why_not_reportable_yet": "",
  "next_action": ""
}
```

### 6.2 二次复现

确认漏洞前必须至少完成两次复现：

1. 第一次：证明异常行为存在。
2. 第二次：证明行为稳定，不是缓存、环境、脏数据、偶发错误。
3. 如涉及权限，必须使用不同角色或不同资源主体交叉验证。
4. 如涉及文件，必须证明文件路径、文件内容、权限边界。
5. 如涉及 JS，必须证明后端接受或信任该前端逻辑。
6. 如涉及依赖漏洞，必须证明项目中存在可达调用链。

### 6.3 失败复现记录

失败复现不能删除，必须记录：

1. 使用的账号。
2. 使用的参数。
3. 请求摘要。
4. 响应摘要。
5. 失败原因。
6. 是否说明漏洞不存在。
7. 是否需要改变假设。

## 7. 工具联动规则

### 7.1 工具选择原则

Claude 可以调用工具，但必须说明：

1. 为什么调用。
2. 输入是什么。
3. 输出是什么。
4. 输出如何改变判断。
5. 是否存在误报。
6. 是否需要人工复核。

不得为了显得复杂而堆砌工具。

### 7.2 推荐工具类型

根据任务选择：

1. `ripgrep / grep`：快速定位路由、危险函数、配置项。
2. AST 工具：提取调用图、import/export、source-to-sink。
3. Semgrep：规则化危险模式发现。
4. CodeQL：跨文件数据流。
5. npm / pnpm / yarn audit：依赖风险。
6. Burp：请求整理、参数对照、角色差异。
7. Playwright：真实业务流、表单提交、按钮状态、前端路由。
8. 真实浏览器：复杂交互、上传、审批、邀请、支付、后台操作。
9. 本地日志：验证服务端真实执行路径。
10. Source Map 工具：还原前端源码。
11. OpenAPI / GraphQL schema：接口枚举与权限对照。
12. 证据 manifest：统一登记证据。

### 7.3 Burp 使用规则

Burp 只用于授权目标：

1. 整理请求。
2. 对比角色。
3. 对比参数。
4. 对比响应。
5. 辅助重放授权请求。
6. 提取 Cookie / Header / CSRF 的合法测试上下文。

不得使用 Burp 对第三方目标进行未授权攻击。

### 7.4 Playwright / 真实浏览器规则

浏览器用于：

1. 登录测试账号。
2. 走完整业务流。
3. 捕获前端路由。
4. 捕获接口请求。
5. 验证按钮、菜单、隐藏入口。
6. 验证多步骤业务。
7. 验证上传、邀请、审批、导出、导入。
8. 生成截图证据。
9. 反哺 Burp 与 JS 分析。

浏览器交互必须遵守最小副作用原则。

## 9. 质量门槛

### 9.1 rejected

满足任一条件应标记为 `rejected`：

1. 已证明是正常业务逻辑。
2. 已证明权限控制有效。
3. 已证明参数不可控。
4. 已证明调用链不可达。
5. 已证明依赖漏洞不可触发。
6. 已证明 Source Map 不造成安全影响。
7. 已证明接口不可访问。
8. 已证明只是错误处理不佳。
9. 已证明只是测试数据。
10. 已证明超出授权范围。

### 9.2 needs_review

满足任一条件应标记为 `needs_review`：

1. 静态证据存在，但动态证据不足。
2. 动态现象存在，但影响不清楚。
3. 工具告警存在，但调用链不清楚。
4. JS 泄露存在，但后端验证不足。
5. 权限差异存在，但业务预期不清楚。
6. 只复现一次。
7. 缺少日志或请求证据。
8. 缺少角色对照。
9. 缺少修复建议。
10. 缺少不可报告原因。

### 9.3 promoted

满足以下条件可标记为 `promoted`：

1. 有明确源码或配置证据。
2. 有初步动态复现。
3. 有合理安全影响。
4. 有下一步验证路径。
5. 没有明显越界。
6. 不是单纯工具告警。

### 9.4 confirmed

满足以下条件可标记为 `confirmed`：

1. 有源码或配置证据。
2. 有动态复现证据。
3. 至少成功复现两次。
4. 有明确影响。
5. 排除了主要误报。
6. 有失败复现记录。
7. 有修复建议初稿。

### 9.5 reportable

满足以下全部条件才可标记为 `reportable`：

1. 有源码或配置证据。
2. 有动态复现证据。
3. 有安全影响证据。
4. 至少成功复现两次。
5. 排除了正常业务逻辑。
6. 排除了测试数据误判。
7. 排除了环境误差。

## 16. 最终原则

1. 没有授权，不测试。
2. 没有证据，不下结论。
3. 没有动态验证，不确认漏洞。

5. 没有二次复现，不标记 confirmed。
6. 没有完整闭环，不标记 reportable。
7. 工具告警不是漏洞。
8. 报错不是漏洞。

10. 猜测不是漏洞。
11. Source Map 不是天然漏洞。
12. JS 泄露不是天然漏洞。
13. 依赖 CVE 不是天然漏洞。
14. 只有可达、可控、可复现、有影响的链路才是漏洞。
